## Context

The mason runtime materializes agent configuration files (`.mcp.json`, `.claude/settings.json`, `.claude/commands/`, `AGENTS.md`, `skills/`) into a `workspace/` directory that is COPY'd into the Docker image via the Dockerfile. The container's WORKDIR is `/home/mason/workspace/project/` (where the user's project is mounted), but these files land at `/home/mason/workspace/` — one level too high. Claude Code never finds them.

The `home/` directory already uses a live bind-mount pattern: files are written to `.mason/docker/{role}/{agent}/home/`, COPY'd into the image as a baked-in fallback, then overlaid with a live mount at `/home/mason/` at runtime. The same pattern needs to apply to `workspace/`.

A parallel problem: `.chapter/acp.json` has been generated by all three materializers since the ACP feature was implemented, but nothing in the codebase reads it at runtime. It is dead output.

## Goals / Non-Goals

**Goals:**
- Claude Code finds `.mcp.json`, `.claude/`, `AGENTS.md`, and `skills/` at the correct path (`/home/mason/workspace/project/`)
- `workspace/` becomes a live bind mount (mirrors the `home/` pattern)
- `build/workspace/project/` files are injected into the project directory via individual bind mount overrides, using the same overlay mechanism already used by `container.ignore.paths`
- `.chapter/acp.json` generation and `generateAcpConfigJson` are removed from all packages
- `agent-launch.json` continues to work without changes (agent-entry hardcodes `/home/mason/workspace/agent-launch.json`)

**Non-Goals:**
- Additive merging of agent files with existing project files (agent files fully shadow overlapping paths — intentional)
- ACP mode changes beyond removing the dead file output
- Changes to `home/` materialization or the `mergeHomeBuild()` mechanism

## Decisions

### Decision: Split materializer output into `workspace/` and `build/workspace/project/`

**Chosen**: Two output buckets keyed by destination:
- `workspace/` keys → written to `.mason/docker/{role}/{agent}/workspace/` → live-mounted to `/home/mason/workspace/`
- `build/workspace/project/` keys → written to `.mason/docker/{role}/{agent}/build/workspace/project/` → mounted as per-file overrides on `/home/mason/workspace/project/`

**Alternative considered**: Keep `workspace/` as-is and adjust the Dockerfile WORKDIR or COPY target. Rejected — the project dir mount happens at runtime and would still shadow any COPY'd files.

**Alternative considered**: Mount the entire `build/workspace/project/` directory over `/home/mason/workspace/project/`. Rejected — this would shadow the entire project directory, defeating the purpose of mounting the user's code.

---

### Decision: Per-file bind mounts for `build/workspace/project/` injection

**Chosen**: Enumerate all files/directories in `build/workspace/project/` and emit individual bind mount entries in docker-compose (same mechanism as sentinel-file masks for `container.ignore.paths`). Directory entries (e.g., `.claude/commands/`) are mounted as directory bind mounts; individual files are mounted as file bind mounts.

**Alternative considered**: Named Docker volumes with pre-populated content. Rejected — more complex to manage and doesn't compose cleanly with the existing project mount.

**Alternative considered**: An `COPY --from` multi-stage build. Rejected — requires image rebuild on every workspace change; live mount is simpler and faster.

---

### Decision: `workspace/` live-mounted, not COPY'd

**Chosen**: The Dockerfile `COPY workspace/` line changes to `COPY build/workspace/`. `workspace/` (containing only `agent-launch.json`) is provided purely via live bind mount at runtime. No baked-in fallback needed — agent-entry will fail explicitly if `agent-launch.json` is missing, which is the correct failure mode.

**Alternative considered**: Keep COPY + live mount (like `home/` does with the backup). Rejected — unlike `home/`, workspace doesn't need OS-level files (`.bashrc` etc.) from the base image. The baked-in fallback adds complexity without benefit.

---

### Decision: Remove `generateAcpConfigJson` entirely

**Chosen**: Delete the function from `agent-sdk/src/helpers.ts`, its export from `agent-sdk/src/index.ts`, and the re-export from `cli/src/materializer/common.ts`. The `acpMode` parameter on `MaterializeOptions` is retained — it is still used by `generateAgentLaunchJson` to select the ACP runtime command.

**Alternative considered**: Keep the function but stop calling it. Rejected — dead exported functions become maintenance debt and confuse future readers.

## Risks / Trade-offs

- **Directory shadow**: Mounting `.claude/commands/` as a directory fully replaces any `commands/` dir in the user's project. If a user has their own slash commands, they'll be hidden. → Acceptable for now; agent commands are intentional overrides. Can revisit with a merge strategy later.

- **Mount count growth**: Each file in `build/workspace/project/` becomes a compose volume entry. A role with many tasks + skills could produce a verbose compose file. → Not a correctness risk; docker-compose handles large mount lists fine.

- **`agent-launch.json` exposure**: With `workspace/` live-mounted, `agent-launch.json` is readable from the host. → Already the case today; no change in exposure.

## Migration Plan

1. Update `docker-generator.ts` to write materializer output to the new paths and add workspace + build overlay mounts to session compose
2. Update `agent-dockerfile.ts` to COPY from `build/workspace/`
3. Update all three materializer `materializeWorkspace()` implementations to use new output key paths
4. Remove `generateAcpConfigJson` from `agent-sdk`, `common.ts`, all materializers, and tests
5. Update all tests — no data migration needed (`.mason/` is generated on demand)

Rollback: revert commits; `.mason/` directories are regenerated on next `chapter run`.
