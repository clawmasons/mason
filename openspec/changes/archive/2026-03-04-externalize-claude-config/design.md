## Architecture

No new modules. This change simplifies the Dockerfile, updates compose volume mounts, and moves config file generation to the install command's file writing phase.

### Dockerfile Changes

The generated Dockerfile is simplified by removing all Claude config/credential setup:

**Before:**
1. Install claude-code
2. Create `.claude.json` with OOBE bypass (heredoc)
3. Create `/home/node/.claude` directory, chown
4. Create entrypoint script for credential injection
5. USER node, WORKDIR, COPY workspace
6. ENTRYPOINT + CMD

**After:**
1. Install claude-code
2. Set DISABLE_AUTOUPDATER
3. USER node, WORKDIR, COPY workspace
4. CMD only (no ENTRYPOINT)

The Dockerfile no longer handles any auth or config — those are provided via volume mounts.

### Volume Mounts (Compose Service)

Two new rw volume mounts are added to the compose service:

- `./claude-code/.claude:/home/node/.claude` — Persistent Claude config directory (credentials written here by `claude /login`)
- `./claude-code/.claude.json:/home/node/.claude.json` — OOBE bypass file

The workspace mount remains: `./claude-code/workspace:/home/node/workspace`

### Install Command Changes

The materializer already generates workspace files. The install command now also writes:

1. `claude-code/.claude.json` — Seeded with OOBE bypass content:
   ```json
   {
     "hasCompletedOnboarding": true,
     "projects": {
       "/home/node/workspace": {
         "hasTrustDialogAccepted": true
       }
     }
   }
   ```

2. `claude-code/.claude/` — Empty directory (created via `mkdirSync`)

These are written alongside the existing workspace files in the install command's file-writing loop. The `.claude.json` is generated by the materializer as a new method.

### Env Template Changes

Remove `claude-code` from `RUNTIME_API_KEYS`. This means:
- `claude-code` runtime no longer adds `CLAUDE_AUTH_TOKEN=` to the env template
- Users authenticate by running `claude /login` inside the container on first run

### User Flow (After Change)

1. `pam install @org/agent-name` — Generates all files including `.claude.json` and empty `.claude/`
2. `docker compose up -d mcp-proxy` — Start proxy
3. `docker compose run --rm claude-code` — Start Claude Code container
4. First run: Claude prompts for login, user authenticates, credentials saved to mounted `.claude/`
5. Subsequent runs: Credentials persist in host-mounted `.claude/` directory
6. Multiple instances: Can share the same `.claude/` mount point

## Decisions

1. **No ENTRYPOINT**: With no credential injection needed, the Dockerfile uses CMD directly. This is simpler and avoids the shell script layer.
2. **RW mounts for .claude**: The `.claude/` directory needs write access for credential storage. `.claude.json` also needs rw since Claude Code may update it.
3. **Materializer generates .claude.json content**: The materializer already knows about the workspace path and OOBE structure, so it provides a `generateClaudeJson()` method. The install command calls it and writes the file.
4. **Empty .claude/ directory**: Created by install command via mkdirSync. No `.gitkeep` needed since the directory is within the generated output.
