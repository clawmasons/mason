## Why

The credential service currently runs as a separate Docker container in the ACP compose stack. This fundamentally doesn't work because the credentials it needs to serve (environment variables, macOS Keychain, `.env` files) live on the host machine, not inside the container. The service resolves credentials from `process.env`, Keychain, and `.env` files - none of which are accessible from within Docker.

The fix is to run the credential service as an in-process SDK within the ACP host process (which already runs on the host), eliminating the Docker container entirely.

## What Changes

- Remove the credential-service container from the Docker Compose stack generated by `AcpSession`
- Run the credential service in-process within the ACP host process using the existing SDK/service layer (`CredentialService`)
- Connect the in-process credential service to the proxy's WebSocket credential relay endpoint (using `CredentialWSClient`)
- Add a credential request to the e2e test (`acp-client-spawn.test.ts`) to verify credentials can be resolved
- Add `TEST_LLM_TOKEN` as a declared credential on the MCP agent in the e2e test chapter config

### Modified Capabilities

## Impact

- **Code**: `packages/cli/src/acp/session.ts` (compose generation, service startup), e2e test files and chapter config
- **Docker**: Credential service container removed from compose stack (fewer containers, simpler networking)
- **Dependencies**: No new dependencies - reuses existing `CredentialService` and `CredentialWSClient` from `packages/credential-service`
- **Runtime**: Credential resolution now happens on the host with direct access to env vars, Keychain, and `.env` files
