## Context

PRD Section 4 defines the project role: an in-memory `Role` generated by scanning the project's agent directory when no `--role` is provided. This is Change 5 in the implementation plan, depending on Changes 1 (Docker pre-flight), 3 (`--source` flag), and 4 (scanner enhancement) — all merged.

The key integration point is `createRunAction()` in `run-agent.ts` (line 622-629), which currently errors when no role is specified. This must be replaced with project role generation. The generated Role then flows into the existing materialization pipeline.

## Goals / Non-Goals

**Goals:**
- Generate an in-memory `Role` from project agent directories when no `--role` is provided
- Pass the generated Role through to mode functions without disk I/O
- Handle error cases per PRD Section 8.3 (missing dir, empty dir, invalid source)
- First-wins deduplication when multiple sources contribute same-named items
- Add container.ignore.paths for source dirs and .env
- Empty instructions (never modify agent system prompt)
- Unit tests for `generateProjectRole()` and the integration in `createRunAction()`

**Non-Goals:**
- Persisting the project role to disk
- Changing existing `--role` flows
- E2E tests (Change 6)

## Decisions

### D1: Pass pre-resolved Role via acpOptions

**Choice:** Add a `preResolvedRole?: Role` field to `runAgent()`'s `acpOptions` parameter. When set, mode functions skip `resolveRoleFn()` and use this Role directly.

**Rationale:** This avoids changing the `runAgent()` function signature (the `role: string` param remains for backward compat). The pre-resolved Role is an optimization path — existing callers continue passing a role name string, and mode functions resolve it as before. Only the project role path sets `preResolvedRole`.

### D2: Generate project role in createRunAction()

**Choice:** In `createRunAction()`, when no role name is available (no `--role`, no alias role), call `generateProjectRole()` to create the Role object, then pass it via `preResolvedRole` to `runAgent()`.

**Rationale:** `createRunAction()` already has access to the project dir, source override, and resolved agent type — all inputs needed for generation. Generating here keeps the Role available before entering mode dispatch.

### D3: Use a sentinel role name for the project role

**Choice:** When generating a project role, pass `"__project__"` as the `role` string parameter to `runAgent()`. Mode functions check for `preResolvedRole` first; the sentinel name is only used for logging/display.

**Rationale:** The `role` parameter is typed as `string` and used for display. A sentinel avoids null/undefined complexity while making it clear this is not a disk-backed role.

### D4: Source directory validation in generateProjectRole()

**Choice:** `generateProjectRole()` checks if each source directory exists before scanning. If none exist, it errors per PRD Section 8.3. If a dir exists but is empty (no tasks, skills, or MCP servers), it warns and proceeds.

**Rationale:** PRD requires specific error messages for missing directories. The validation must happen before `scanProject()` since the scanner silently skips non-existent directories.

### D5: First-wins deduplication for multi-source

**Choice:** When multiple sources are specified, iterate sources in order. For each item type (tasks, skills, apps), skip items whose name already exists from an earlier source.

**Rationale:** PRD Section 4.2 specifies first-wins deduplication. The scanner returns items ordered by source (it iterates dialects in order), so we deduplicate by name using a Set.

## Implementation

### generateProjectRole() function

Location: `packages/cli/src/cli/commands/run-agent.ts` (exported)

```typescript
export async function generateProjectRole(
  projectDir: string,
  sources: string[],    // dialect registry keys (e.g., ["claude-code-agent"])
  agentType: string,    // target agent type for source.agentDialect
): Promise<Role>
```

Steps:
1. For each source, resolve the directory name from the dialect registry
2. Check each source directory exists (`fs.existsSync(join(projectDir, `.${dir}/`))`)
3. If NO source directory exists, `console.error()` per PRD 8.3 and `process.exit(1)`
4. Call `scanProject(projectDir, { dialects: sources })`
5. Deduplicate results by name (first-wins)
6. Map `ScanResult` → `Role`:
   - `commands` → `tasks` (as `TaskRef[]`)
   - `skills` → `skills` (as `SkillRef[]`)
   - `mcpServers` → `apps` (as `AppConfig[]`)
7. Build `container.ignore.paths` from source directories + `.env` if present
8. Return the Role object

## Test Coverage

### Unit tests for generateProjectRole()

1. **Single source with tasks, skills, apps:** Create temp dir with `.claude/commands/review.md`, `.claude/skills/test/SKILL.md`, `.claude/settings.json` with MCP server. Verify Role has correct tasks, skills, apps arrays.
2. **Multi-source first-wins dedup:** Create `.claude/commands/review.md` and `.codex/instructions/review.md`. Verify only the claude version appears.
3. **Missing source directory:** Call with non-existent source, expect `process.exit(1)`.
4. **Empty source directory:** Create `.claude/` with no tasks/skills/servers. Verify Role is returned with empty arrays and a warning is logged.
5. **Container ignore paths:** Verify `.claude/` is in ignore paths. Verify `.env` added when present, absent when not.
6. **Instructions always empty:** Verify `role.instructions === ""`.
7. **Metadata:** Verify `metadata.name === "project"`, `type === "project"`.

### Integration test for createRunAction() project role path

1. **No --role triggers project role:** Mock `runAgent`, invoke action without `--role`, verify `preResolvedRole` is passed.
