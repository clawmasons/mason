## Context

The forge proxy Dockerfile is generated by `generateProxyDockerfile()` in `src/generator/proxy-dockerfile.ts` and used by `runInstall()` in `src/cli/commands/install.ts`. Currently, the Dockerfile uses a multi-stage build that compiles forge from source. With the forge-packaging changes (Changes 1-3), forge is now an npm package (`@clawmasons/forge`) that users install via `npm install`. The built artifacts are already available in `node_modules/@clawmasons/forge/dist/`, making source compilation in Docker unnecessary.

## Goals / Non-Goals

**Goals:**
- Replace multi-stage Dockerfile with a single-stage build using pre-built forge
- Update `runInstall()` to copy pre-built forge artifacts instead of source files
- Update all related tests to match new expectations
- Maintain the same runtime behavior (forge proxy starts correctly)

**Non-Goals:**
- Changing the Docker compose generation (already correct -- uses `build: ./forge-proxy`)
- Changing the workspace directory copying logic (unchanged)

## Decisions

### 1. Single-stage Dockerfile with production dependency install

The new Dockerfile uses a single `FROM node:22-slim` stage. Package manifests (`package.json`, `package-lock.json`) are copied first for Docker layer caching, then `npm ci --omit=dev --ignore-scripts` installs production dependencies. Pre-built `dist/` and `bin/` are copied after. The workspace is copied as `workspace/` under `/app/`.

This approach avoids copying the entire `node_modules/` tree (which is enormous during development and contains dev dependencies + binary files that can't be read as UTF-8 text). Instead, Docker handles dependency installation from the lockfile, producing a minimal production-only `node_modules/`.

### 2. Entrypoint changes from /app/bin/forge.js to /app/forge/bin/forge.js

Since the forge package is now nested under `/app/forge/` instead of being at `/app/` root, the entrypoint path is updated to `/app/forge/bin/forge.js`.

### 3. runInstall copies dist, bin, package.json, package-lock.json only

The `runInstall()` function uses `getForgeProjectRoot()` to locate the forge installation. It copies only `dist/`, `bin/`, `package.json`, and `package-lock.json` into `forge-proxy/forge/`. The old source file copies (`src/`, `tsconfig*.json`) and the full `node_modules/` copy are removed.

### 4. copyDirToFiles gains configurable skipDirs parameter

The `copyDirToFiles()` helper now accepts an optional `skipDirs` parameter (default: `["node_modules", ".git"]`). For forge `dist/` and `bin/` copying, only `[".git"]` is passed as skipDirs, allowing the full directory trees to be copied. For workspace directory copying, the default `["node_modules", ".git"]` is used.

## Risks / Trade-offs

- **[Acceptable] Docker build requires network for npm ci** -- The Dockerfile runs `npm ci --omit=dev` which requires network access during build. This is standard Docker practice and the command is fast (production deps only, no scripts).
- **[Risk] getForgeProjectRoot() accuracy** -- This function traverses from the current file's location. When forge is installed in node_modules, it correctly resolves to the forge package root. Same behavior as before.
