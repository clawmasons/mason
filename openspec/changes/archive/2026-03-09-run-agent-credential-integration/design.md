## Context

The `run-agent` command currently generates a docker-compose.yml with two services (proxy and agent). The agent container receives all LLM provider API keys via environment variables. With the credential service infrastructure now in place (CHANGE 3, 4), we need to route credentials through the credential service instead.

The `ResolvedAgent` type already has `credentials: string[]` and `ResolvedRole` has `risk: "HIGH" | "MEDIUM" | "LOW"` from CHANGE 1. The `ResolvedApp` also has `credentials: string[]`. The credential service Dockerfile is generated by `docker-init` (CHANGE 7).

## Goals / Non-Goals

**Goals:**
- Display required credentials and risk level before container launch
- Generate CREDENTIAL_PROXY_TOKEN for credential service auth
- Add credential-service container to docker-compose with proper dependency chain
- Remove API keys from agent container environment
- Maintain backward compatibility for the compose generation API

**Non-Goals:**
- Resolving credential values (that's the credential service's job at runtime)
- Validating credential availability (that's `chapter validate`, CHANGE 10)
- Agent entry bootstrap flow (CHANGE 6)

## Decisions

### Decision 1: Credential resolution uses agent.credentials union with app.credentials from the role

**Choice**: Collect `agent.credentials` plus all `app.credentials` from the role's apps. Display each credential with its declaring package(s).

**Rationale**: This matches the PRD UC-5 display format. The union gives the operator full visibility into what credentials will be requested at runtime.

### Decision 2: generateComposeYml gains credentialProxyToken parameter

**Choice**: Add `credentialProxyToken` to the opts parameter of `generateComposeYml`. The credential-service service is always generated.

**Rationale**: The credential service is unconditionally generated by docker-init (Decision 4 in CHANGE 7), so run-agent should unconditionally include it in compose. The token is generated alongside the proxy token.

### Decision 3: Agent environment only has MCP_PROXY_TOKEN

**Choice**: Remove the `PROVIDER_ENV_VARS` loop from compose generation. The agent service environment has only `MCP_PROXY_TOKEN=${proxyToken}` (renamed from CHAPTER_PROXY_TOKEN for clarity at the agent level).

**Rationale**: Per REQ-021, no API keys should appear in agent container environment. Credentials flow through credential service at runtime.

### Decision 4: Credential service Dockerfile path follows docker-init convention

**Choice**: The credential service Dockerfile is at `credential-service/Dockerfile` relative to the docker build context, matching what docker-init generates.

**Rationale**: Consistency with CHANGE 7. The compose file references the same path docker-init writes to.

### Decision 5: Proxy also receives CREDENTIAL_PROXY_TOKEN

**Choice**: Pass CREDENTIAL_PROXY_TOKEN to both the proxy and credential-service containers.

**Rationale**: Per PRD section 7.2, the proxy needs this token to authenticate WebSocket connections from the credential service.

### Decision 6: Credential display is informational only

**Choice**: Display credentials and risk level via console.log but don't block on user confirmation.

**Rationale**: The PRD UC-5 shows an informational display. Blocking would break scripted/automated usage. Validation is separate (CHANGE 10).

## API Surface

```typescript
// Updated generateComposeYml opts
interface ComposeOpts {
  dockerBuildPath: string;
  projectDir: string;
  agent: string;
  role: string;
  logsDir: string;
  proxyToken: string;
  credentialProxyToken: string;  // NEW
}

// New helper functions
function resolveRequiredCredentials(agent: ResolvedAgent, role: ResolvedRole): Map<string, string[]>;
function displayCredentials(credentials: Map<string, string[]>, riskLevel: string): void;
```

## Dependency Chain in Compose

```
proxy-<role>
  |
  +-- credential-service (depends_on: proxy)
        |
        +-- agent-<agent>-<role> (depends_on: credential-service)
```
