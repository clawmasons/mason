## Context

Existing e2e tests in `mason-extensions` validate the full agent pipeline (Docker build, agent execution, prompt responses, MCP tool usage) but skip verification of the materialized workspace — the files generated by each agent's materializer before the container starts. This means a materializer regression (e.g., wrong path for skills, missing MCP config) would only surface as a cryptic agent failure, not a clear "file missing" assertion.

The `claude-test-project` fixture already provides tasks, skills, and MCP config in claude format. The existing e2e tests already run `mason run` with this fixture. The materialized files persist on disk at `.mason/docker/<role>/<agent>/` after each run.

## Goals / Non-Goals

**Goals:**
- Verify that tasks, skills, and MCP config are materialized to the correct agent-specific paths after `mason run`
- Cover the native-source case (claude → claude) and cross-source cases (claude → codex, claude → pi)
- Provide a reusable helper in `@clawmasons/agent-sdk/testing` for artifact path checking
- Integrate into existing `test:e2e` CI with no new infrastructure

**Non-Goals:**
- Testing materializer logic in isolation (already covered by unit tests in each agent's `materializer.test.ts`)
- Testing artifact _content correctness_ beyond basic string matching (unit tests handle that)
- Adding new fixtures — reuse `claude-test-project`
- Testing agents not in `mason-extensions` (e.g., the built-in `mcp-agent`)

## Decisions

### 1. Add artifact checks to existing e2e tests vs. separate test files

**Decision**: Separate `artifacts.test.ts` files per agent.

**Why**: The existing tests have 300s timeouts and require API keys + Docker. Artifact checks only need Docker (no API key) since we can use `--build` mode or check artifacts after a regular run. Separate files allow running artifact verification independently and keep concerns clean.

**Alternative considered**: Adding `expect()` calls inside existing test cases. Rejected because it couples artifact verification to prompt execution — if the prompt fails (API issue), artifact checks are skipped even though materialization succeeded.

### 2. Where to check: build directory vs. container filesystem

**Decision**: Check `.mason/docker/<role>/<agent>/` directories on the host filesystem.

**Why**: These directories are populated during materialization (before Docker starts the container). They persist after the test run. No need to exec into a container or inspect Docker volumes.

Three subdirectories matter:
- `build/workspace/project/` — project file overlays (tasks, skills, settings)
- `home/` — agent home config (`.claude.json`, `.codex/config.toml`, etc.)
- `workspace/` — `agent-launch.json`

### 3. Helper API design

**Decision**: Add `testWorkspaceArtifacts()` to `@clawmasons/agent-sdk/testing` that takes a flat check list.

```ts
type ArtifactCheck = { path: string; contains?: string };

function testWorkspaceArtifacts(
  workspaceDir: string,
  role: string,
  agentType: string,
  checks: {
    buildFiles?: ArtifactCheck[];
    homeFiles?: ArtifactCheck[];
    workspaceFiles?: ArtifactCheck[];
  },
): void
```

**Why**: Follows the pattern of existing helpers (`testFileContents`, `testSessionLogContains`) — throws descriptive errors on failure rather than returning booleans. The `role`/`agentType` parameters construct the base path (`.mason/docker/{role}/{agentType}/`) so tests don't repeat path math.

**Alternative considered**: Per-agent helpers (e.g., `testClaudeArtifacts`). Rejected because the path structure is the same across agents — only the file list differs, which the caller provides.

### 4. Test execution strategy

**Decision**: Each `artifacts.test.ts` runs its own `mason run` command (via `runMasonPrint`) to trigger materialization, then checks the build directory.

**Why**: This is a true e2e test — it exercises the full pipeline from CLI invocation through role resolution, source normalization, materializer dispatch, and file writing. The existing e2e pattern already handles workspace setup, Docker availability checks, and cleanup.

The first test case in each file can use a simple prompt (e.g., "what is 2+2") since we only need materialization to complete — the agent's response doesn't matter for artifact verification.

## Risks / Trade-offs

**[Docker required]** → Same as existing e2e tests. Tests skip gracefully when Docker is unavailable.

**[API key required for full run]** → Artifact checks piggyback on a real `mason run` which needs an API key. Mitigation: we could explore a `--dry-run` or `--materialize-only` flag in the future, but for now we match the existing e2e pattern.

**[Build dir path coupling]** → Tests depend on the `.mason/docker/<role>/<agent>/` directory structure. If this changes, the helper needs updating. Mitigation: the helper centralizes path construction, so only one place to update.

**[Fixture content coupling]** → Tests assert specific strings from the `claude-test-project` fixture (e.g., `"Take Notes"`, `"Markdown Conventions"`). If the fixture changes, tests break. Mitigation: this is intentional — the fixture is the contract, and breakage means a reviewer must update both.
