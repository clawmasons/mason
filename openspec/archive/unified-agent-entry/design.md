## Goals

1. Make `agent-entry` the single entrypoint for all agent containers via `agent-launch.json` config
2. Support file-based credential installation (Claude Code needs `~/.claude/.credentials.json`)
3. Sync SSE parsing fixes from mcp-agent to agent-entry MCP client
4. Add `security.*` allowlist support in credential service (rejects unknown keys with ACCESS_DENIED)
5. Add bash-agent as a new agent type
6. Unify all agent Dockerfiles to use `agent-entry` entrypoint

## Decisions

### agent-launch.json schema
```json
{
  "credentials": [
    { "key": "CLAUDE_CODE_OAUTH_TOKEN", "type": "env" },
    { "key": "OPENROUTER_API_KEY", "type": "env" }
  ],
  "command": "claude",
  "args": []
}
```
- `credentials[].type`: `"env"` (set as env var) or `"file"` (write value to `path`)
- `command` + `args`: replaces `AGENT_RUNTIME_CMD` env var
- `MCP_PROXY_TOKEN` and `MCP_PROXY_URL` remain as env vars (set by Docker Compose)
- `AGENT_CREDENTIALS` env var is kept as fallback but `agent-launch.json` takes precedence
- `agent-launch.json` is generated by materializers and COPY'd into the Docker image

### Credential service security.* handling
- Hardcoded allowlist: currently empty, reserved for future security-sensitive credentials
- Any `security.*` key returns ACCESS_DENIED (none are currently permitted)
- This is handled in the resolver before the normal env/keychain/dotenv chain

### Unified Dockerfile entrypoint
- All agent Dockerfiles switch to `ENTRYPOINT ["agent-entry"]`
- `agent-entry` binary is available via `node_modules/.bin/agent-entry` (already in PATH from `/app/node_modules/.bin`)
- Per-runtime install steps (npm install -g) remain unchanged
- ACP mode: materializer generates different `agent-launch.json` with ACP command

### bash-agent materializer
- Minimal materializer: generates only `agent-launch.json` and `AGENTS.md`
- Same credential setup as claude-code (fetches `CLAUDE_CODE_OAUTH_TOKEN` as env var)
- Command: `bash` (no args)
- Registered in materializer registry as "bash-agent"

## Risks

- **Breaking change**: Existing Dockerfiles change entrypoint from runtime-specific to `agent-entry`. Mitigated by keeping `AGENT_RUNTIME_CMD` as fallback.
- **File permissions**: Writing credential files inside container requires directory to exist. Mitigated by `mkdir -p` before write.
- **SSE parsing**: If proxy changes response format, both mcp-agent and agent-entry need updating. Mitigated by keeping implementations aligned.

## Trade-offs

- **Config file vs env vars**: `agent-launch.json` adds a file to manage but enables file-based credentials and complex configs that env vars can't express cleanly.
- **Allowlist vs generic keychain**: Restricting to allowlisted keys is more secure but requires code changes to add new keychain credentials. This is intentional — security credentials should require explicit code review.
