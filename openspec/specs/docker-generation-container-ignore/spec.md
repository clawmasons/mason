# docker-generation-container-ignore Specification

## Purpose

Role-centric Docker build directory generation and container ignore (volume masking) for the ROLE_TYPES pipeline. Generates Docker build directories at `.mason/docker/<role-name>/` matching PRD section 7.1 structure, implements volume masking for `container.ignore.paths`, creates sentinel files, and generates self-contained session directories.

## Requirements

### Requirement: Role-centric Docker build directory structure

The system SHALL generate a Docker build directory at `.mason/docker/<role-name>/` with the following structure:
- `<agent-type>/Dockerfile` — Agent Dockerfile generated by `generateAgentDockerfile()`
- `<agent-type>/workspace/` — Materialized workspace files from `materializeForAgent()`
- `mcp-proxy/Dockerfile` — MCP proxy Dockerfile using `node:22-slim` base image
- `docker-compose.yaml` — Reference compose file (informational only)

#### Scenario: Build directory created for a role
- **GIVEN** a valid `Role` with agent type "claude-code-agent"
- **WHEN** `generateRoleDockerBuildDir()` is called
- **THEN** the directory `.mason/docker/<role-name>/claude-code-agent/Dockerfile` SHALL exist
- **AND** the directory `.mason/docker/<role-name>/claude-code-agent/workspace/` SHALL contain materialized files
- **AND** the file `.mason/docker/<role-name>/mcp-proxy/Dockerfile` SHALL exist and use `node:22-slim`
- **AND** the file `.mason/docker/<role-name>/docker-compose.yaml` SHALL exist

### Requirement: Volume masking for container.ignore.paths

The system SHALL implement volume masking to hide specified project paths inside the container's project mount at `/home/mason/workspace/project/`.

- Paths ending with `/` SHALL be classified as directories and masked with Docker named empty volumes.
- Paths without trailing `/` SHALL be classified as files and masked with read-only bind mounts of a sentinel empty file.
- Volume names SHALL be sanitized from the path (prefix `ignore-`, non-alphanumeric characters replaced with hyphens).

#### Scenario: Directory path is masked with named volume
- **GIVEN** `container.ignore.paths` contains `.mason/`
- **WHEN** volume masks are generated
- **THEN** a named volume `ignore-mason` SHALL be mounted at `/home/mason/workspace/project/.mason`

#### Scenario: File path is masked with sentinel bind mount
- **GIVEN** `container.ignore.paths` contains `.env`
- **WHEN** volume masks are generated
- **THEN** the sentinel empty file SHALL be bind-mounted read-only at `/home/mason/workspace/project/.env`

### Requirement: Volume masking targets only project mount paths

Volume masking SHALL apply exclusively to the project mount at `/home/mason/workspace/project/`. The materialized workspace at `/home/mason/workspace/` (containing agent configuration files like `.claude/settings.json`) SHALL remain unaffected.

#### Scenario: Materialized workspace is not masked
- **GIVEN** `container.ignore.paths` contains `.claude/`
- **WHEN** the session compose file is generated
- **THEN** the mask SHALL target `/home/mason/workspace/project/.claude` only
- **AND** `/home/mason/workspace/.claude/` (materialized workspace) SHALL NOT be masked

### Requirement: Sentinel file creation

The system SHALL create a sentinel empty file at `.mason/empty-file` with permissions `0o444` (read-only for all). The creation SHALL be idempotent — if the file already exists, no action is taken.

#### Scenario: Sentinel file created
- **WHEN** `ensureSentinelFile()` is called for the first time
- **THEN** `.mason/empty-file` SHALL be created with 0 bytes and mode `0o444`

#### Scenario: Sentinel file already exists
- **WHEN** `ensureSentinelFile()` is called and the file already exists
- **THEN** no file operations SHALL be performed

### Requirement: Session directory as functional Docker Compose project

The system SHALL create session directories at `.mason/sessions/<session-id>/` containing:
- `docker-compose.yaml` — A self-contained compose file with all paths relative to the session directory
- `logs/` — Directory for session logs

All build contexts, volume mounts, environment variables, and sentinel file references SHALL be resolvable from the session directory using relative paths. Users SHALL be able to run `docker compose logs`, `docker compose ps`, `docker compose exec`, and `docker compose down` directly from the session directory.

#### Scenario: Session compose file uses relative paths
- **WHEN** a session directory is created
- **THEN** the compose file SHALL NOT contain absolute host paths
- **AND** build contexts SHALL use relative paths from session dir to Docker build dir
- **AND** volume mounts SHALL use relative paths from session dir to project root

#### Scenario: Session compose file includes volume masks
- **GIVEN** a role with `container.ignore.paths: [".mason/", ".claude/", ".env"]`
- **WHEN** a session is created
- **THEN** the compose file SHALL include named volumes for `.mason/` and `.claude/`
- **AND** the compose file SHALL include a sentinel bind mount for `.env`
- **AND** the compose file SHALL include a `volumes:` section declaring named volumes

### Requirement: Proxy Dockerfile uses @clawmasons/proxy from pre-populated node_modules

The proxy Dockerfile in `mcp-proxy/Dockerfile` SHALL:
- Use `node:22-slim` as the base image
- Copy `node_modules/` from the `docker/` build context (pre-populated by `docker-init`)
- Run as `USER mason`
- Execute `mason proxy --agent <agentName> --transport streamable-http` as the entrypoint

#### Scenario: Proxy Dockerfile content
- **WHEN** a proxy Dockerfile is generated
- **THEN** it SHALL contain `FROM node:22-slim`
- **AND** it SHALL contain `COPY node_modules/`
- **AND** it SHALL contain `USER mason`
- **AND** the CMD SHALL reference `mason` and `proxy`
