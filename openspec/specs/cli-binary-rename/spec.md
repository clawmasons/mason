## Purpose

Rename the CLI binary, program name, and all internal references from `chapter` to `mason`. This is the foundational change for the mason-cli PRD, aligning the binary name with the product brand. All existing commands continue to work as top-level commands â€” no restructuring yet.

## Requirements

### Requirement: Binary entry point renamed to `mason`
The system SHALL register the CLI executable as `mason` in `packages/cli/package.json`'s `bin` field. The bin script file SHALL be named `bin/mason.js`. The package SHALL also be published as `clawmasons` and `clawmason` (unscoped aliases).

#### Scenario: CLI is invocable as `mason`
- **WHEN** the user runs `npx mason --help`
- **THEN** the CLI outputs usage information listing all registered commands

#### Scenario: `clawmasons` binary no longer resolves
- **WHEN** the user runs `npx clawmasons --help`
- **THEN** the command fails because the `clawmasons` bin entry no longer exists

### Requirement: Commander program name updated
The system SHALL set the Commander program name to `mason` in `packages/cli/src/cli/index.ts`.

#### Scenario: Help output shows correct program name
- **WHEN** the user runs `npx mason --help`
- **THEN** the usage line reads `Usage: mason [options] [command]`

### Requirement: Proxy Dockerfile entrypoint updated
The system SHALL update the generated proxy Dockerfile entrypoint to reference `mason` instead of `clawmasons`.

#### Scenario: Generated proxy Dockerfile uses new binary name
- **WHEN** `mason build` generates a proxy Dockerfile
- **THEN** the ENTRYPOINT line references `node_modules/.bin/mason`
- **AND** the CMD line still passes `proxy --agent <agentName>`

### Requirement: E2E test helpers updated
The system SHALL update `packages/tests/tests/helpers.ts` to reference `bin/mason.js` instead of `bin/clawmasons.js` for the `CLAWMASONS_BIN` constant (renamed to `MASON_BIN`).

#### Scenario: E2E tests use new binary path
- **WHEN** E2E tests execute CLI commands
- **THEN** they invoke `bin/mason.js`

### Requirement: User-facing CLI messages updated
The system SHALL update all user-facing log messages, help text, error messages, and "Generated by" comments that reference the `clawmasons` binary name to use `mason` instead. Domain-concept references to "chapter" (e.g., "chapter workspace", "chapter package", the `"chapter"` field in JSON configs) are NOT renamed.

#### Scenario: Build output shows mason in next-steps
- **WHEN** `mason build` completes successfully
- **THEN** the "next steps" output references `mason run-agent` and `mason run-acp-agent`

#### Scenario: ACP agent log prefix uses mason
- **WHEN** the ACP agent logs messages
- **THEN** the log prefix reads `[mason run-acp-agent]`

#### Scenario: Generated Dockerfiles reference mason
- **WHEN** Dockerfiles are generated by `docker-init`
- **THEN** the header comment reads `# Generated by mason docker-init`

### Requirement: E2E test assertions updated
The system SHALL update E2E test assertions that check for the `clawmasons` binary name in generated artifacts (Dockerfiles, .bin symlinks) to check for `mason` instead.

#### Scenario: Build pipeline test checks for mason binary
- **WHEN** the build pipeline E2E test checks `docker/node_modules/.bin/`
- **THEN** it asserts a `mason` symlink exists (not `clawmasons`)

#### Scenario: Proxy Dockerfile test checks for mason
- **WHEN** the build E2E test checks the generated proxy Dockerfile
- **THEN** it asserts the Dockerfile contains `mason` (not `clawmasons`)

### Requirement: Unit test assertions updated
The system SHALL update all unit test assertions that check for the `clawmasons` binary name in CLI output, generated files, and log messages to check for `mason` instead.
